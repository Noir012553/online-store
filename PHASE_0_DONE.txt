================================================================================
KEY FINDINGS FROM ANALYSIS
================================================================================

BOTTLENECK #1: N+1 Query (CRITICAL) 🔴
├─ Problem: Each spec stored as separate document
│          Product has 20 specs = 26 documents (1 name + 1 desc + 1 brand + 20 specs + 3 features)
│          1000 products = 26,000 documents
├─ Current: Query time = 500-2000ms per product
└─ Solution: Specs as Map → 1 query = 50-100ms (10-20x faster) ✅

BOTTLENECK #2: Layout Shift (CRITICAL) 🔴
├─ Problem: setLocale() clears entire translation cache
│          UI shows raw keys: "footer.description" while loading
├─ Current: Blinky/broken screen experience
└─ Solution: SWR pattern (keep old, load new async, flip when done) ✅

BOTTLENECK #3: No Rate Limiting (CRITICAL) 🔴
├─ Problem: Unlimited API calls to Cloudflare AI
│          When 429 happens → service crashes
├─ Current: No recovery mechanism
└─ Solution: Queue + Throttling + Exponential Backoff ✅

BOTTLENECK #4: No Audit Trail (HIGH) 🟠
├─ Problem: Admin edits translations but no log
│          Can't track: Who changed? When? Old value?
├─ Current: Compliance nightmare
└─ Solution: TranslationAuditLog with full context ✅

BOTTLENECK #5: Uniform TTL (HIGH) 🟠
├─ Problem: 30-day TTL for all data is wrong
│          Product specs deleted too soon
│          Review data kept too long
└─ Solution: Per-collection TTL (Product: 90d, Review: 30d) ✅

BOTTLENECK #6: Monolithic Translation Load (MEDIUM) 🟡
├─ Problem: Load entire file (50KB+) even if only need 5KB
├─ Current: LCP penalty, slower initial load
└─ Solution: Route-based lazy loading ✅

BOTTLENECK #7: No Offline Support (MEDIUM) 🟡
├─ Problem: Works only when online
└─ Solution: IndexedDB caching for offline ✅

BOTTLENECK #8: No Idempotency Lock (MEDIUM) 🟡
├─ Problem: Duplicate requests = duplicate API calls
└─ Solution: Idempotency lock + pending status ✅
